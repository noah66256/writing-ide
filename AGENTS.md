# AGENTS.md（本仓库 AI 编程助手规则）

> 本文件用于兼容更多 AI coding agents 的“记忆文件”标准（参考 [AGENTS.md](https://agents.md/)）。
> 内容与仓库内的 `.cursorrules` 保持一致。

你是本仓库的 AI 编程助手（Cursor）。

## 产品定位（不可偏离）
我们在做一款「写作 IDE（桌面端）」：左侧文件树，中间编辑器（Tab+补全），右侧 Agent（Plan/Agent/Chat），中下方 Dock Panel（KB/Outline/Graph/Problems/Search/Runs）。
一切功能都必须服务于“写作产出与编辑体验”，避免跑偏成工作流平台/通用协作平台。

## 工程方法论（强制 SOP：需求 → 范式 → 大搜 → 计划 → 落地 → 验证）
这套流程是默认路径；除非用户明确说“先不做/只聊/跳过”，否则必须按此推进。

### 0) 触发
- 用户提出需求、报告 bug、或表达“不满足预期”。

### 1) 对齐需求（最多 5 个高价值澄清）
- 先复述：目标 / 成功标准 / 约束 / 不做什么。
- 澄清上限：最多 5 个高价值问题。
  - 写作任务优先：平台画像 / 受众 / 目标 / 风格 / 素材来源。
  - 工程问题优先：复现条件 / 期望行为 / 现象截图或日志 / 影响范围 / 回滚要求。

### 2) 范式优先（先找实现方法，不做零散修补）
- **先抽象“系统性机制/范式”**：例如 Gate/Router（门禁/路由）、Contract（契约/可见性/权限边界）、State Machine（状态机）、Registry（注册表/能力目录）、Cache（缓存）、Fallback（降级/回退）、Observability（可观测性）、Undo/Rollback（可撤销）。
- 目标是找到可复用的解法，而不是只修一处 if/else 或单点补丁。

### 3) 全网 + GitHub 大搜（研究先行）
- 当问题涉及“机制/范式选择”或出现反复失败/重试/误伤时，必须做“大搜”：
  - 搜索主流方案、对比优缺点、适用条件与常见坑位。
  - 给出“对我们项目最合适的落地版本”（先最小闭环，再扩展）。
- **研究产物必须落盘到仓库 md**（新建或更新），并在 `plan.md` 的“导航（常用入口）”补链接，保证可检索、可复刻。

### 4) 落到本项目（Plan 驱动）
- 每次新对话开头必须阅读根目录 `plan.md` 与 `README.md`，确保不偏离当前决策。
- 在开始实现前，把本轮任务拆成可执行 todo（用 Cursor 的 todo 工具或在回复里列清）。
- 若新增机制/协议/策略：先写工程说明 md（v0.x，含验收/回滚）再落代码；并把入口补进 `plan.md`。

### 5) 实现（proposal-first）
- 任何“写入文件/批量修改”必须先提供可审阅的 diff/提案，再执行（涉及用户内容更要谨慎）。
- 外部依赖/脚手架命令必须非交互（`--yes` 等）。
- 变更必须可回滚：Keep/Undo、proposal-first、或显式回滚方案。

### 6) 验证与复盘（闭环）
- 必须给出“如何验证”的 checklist（本地/线上/复现步骤）。
- 若涉及线上部署：必须包含健康检查与关键日志关键词（例如 `policy.decision` 的 `derivedFrom/routeId/clarify`）。
- 对常见坑：沉淀到 `debug.md`；对方案：更新对应 spec，并在 `plan.md` 维护入口。

## 每次新对话的固定流程（必须遵守）
1. 先阅读根目录 `plan.md` 与 `README.md`，确保不偏离当前决策。
2. 在开始实现前，把本轮任务拆成可执行 todo（用 Cursor 的 todo 工具或在回复里列清）。
3. 优先按里程碑从小到大推进：先打通骨架与最小闭环，再扩展。
4. 任何“写入文件/批量修改”必须先提供可审阅的 diff/提案，再执行（如果涉及用户内容更要谨慎）。
5. 任何涉及外部依赖/脚手架命令，必须使用非交互参数（--yes 等）。

## 产品关键约束（不要忘）
- Agent 由 LLM 自主决定调用哪些工具；系统不写死链式流程。
- 澄清策略：默认先问最多 5 个高价值问题（平台画像/受众/目标/风格/素材来源）。
- 右侧模式：
  - Plan：逐步产出（可确认/回退）
  - Agent：一次成型+对话迭代（仍可随时打断）
  - Chat：纯对话，无文件编辑权限（多模态输入保留）
- 右侧输出形态：流式输出 + 工具卡片（Tool Blocks）；每个工具步骤模块化展示并支持 Keep/Undo（Undo 需可撤销或走“提案→确认→执行”）。
- Plan/Agent 模式必须维护“主文档（逻辑主线）”，每次调用模型前强制注入，防上下文跑偏（主文档默认按每个 Run/对话维护）。
- 产品内应提供 Doc Rules（文档规则，Markdown，可在设置里编辑）的项目级规则文件/面板，跨 Run 生效，作为长期约束与风格章程（同样纳入 Context Pack）。
- 工具协议：Schema（校验）+ XML（可解析外壳），工具调用消息不得夹杂自然语言。
- KB：Postgres+pgvector；检索默认入口是 outline，并支持 outlineFacet；结果按 source_doc 分组去重。
- B 端：admin-web 仅做账号/权限、LLM 配置（热生效）、工具/Run 审计等管理能力。
- 计费：以充值积分为主；后续按模型调用 usage 扣费并可审计。

## 代码组织约定
- `apps/desktop`：C 端桌面客户端
- `apps/gateway`：统一后端/代理（Auth/Models/Tools/KB/Content Lab/Config hot-reload）
- `apps/admin-web`：B 端管理后台
- `packages/*`：共享类型、Agent 核心、工具、KB 核心算法等

## 语言与输出
- 始终使用简体中文回复。
- 结论先行，避免长篇赘述；给出明确的下一步与需用户确认点。

## 开发约定：终端默认 Git Bash（Windows）
- 默认终端：Git Bash（`bash.exe`）。
- 不要切换 PowerShell/CMD 来执行命令（避免行为与语法不一致）。
- 必须调用 Windows 原生命令时：在 Git Bash 里用 `cmd.exe /c <command>` 调用，避免 Git Bash 的路径转换（例如把 `/PID` 误当路径）。
