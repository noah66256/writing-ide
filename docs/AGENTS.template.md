## 工程方法论（强制 SOP：需求 → 范式 → 大搜 → 计划 → 落地 → 验证）
这套流程是默认路径；除非用户明确说“先不做/只聊/跳过”，否则必须按此推进。

### 1) 对齐需求（最多 5 个高价值澄清）
- 先复述：目标 / 成功标准 / 约束 / 不做什么。
- 若信息不足：最多问 5 个问题，优先补齐：
  - **target**（要改哪里：模块/文件/选区/接口）
  - **action**（要做什么：实现/修复/改写/解释/对比）
  - **acceptance**（怎么验收：用例/截图/日志/行为）
  - **risk/rollback**（失败怎么回退：开关/降级/回滚）
  - **permission**（是否允许：改文件/跑命令/联网/部署）

### 2) 范式优先（先找实现方法，不做零散修补）
- 先抽象“系统性机制/范式”，优先给可复用解法：Gate/Router、Contract、State Machine、Registry、Cache、Fallback、Observability、Undo/Rollback 等。
- 目标是找到**可扩展、可维护、可观测、可回滚**的方案，而不是只修一个 if/else。

### 3) 全网 + GitHub 大搜（研究先行）
- 当问题涉及“机制/范式选择”、或出现反复失败/误伤时，先做大搜：
  - 搜主流方案、对比优缺点、适用条件、常见坑位。
  - 选一个“对当前项目最合适的落地版本”（先最小闭环，再扩展）。
- 研究产物必须落盘为 `docs/research/<topic>.md`（或等价路径），并在 `plan.md` 补入口，保证可检索、可复刻。

### 4) 落到项目本身（Plan 驱动）
- 每次开始实现前，先写/更新 `plan.md`（或 `docs/plan.md`）：
  - 背景/目标
  - 方案与权衡
  - 里程碑/拆分 todo
  - 验收/回滚
- 复杂任务必须拆 todo；一次只推进一个最小闭环，避免“大而糊”。

### 5) 实现（proposal-first）
- 任何“写入文件/批量修改”必须先给可审阅的 diff/提案，再执行。
- 变更必须可回滚（保留 Undo/回滚策略、或以 feature flag/开关保护）。
- 外部依赖/脚手架命令必须非交互（`--yes` 等）。

### 6) 验证与复盘（闭环）
- 必须给出“如何验证”的 checklist（本地/线上/复现步骤）。
- 若涉及部署：必须包含健康检查与关键日志关键词。
- 常见坑沉淀到 `debug.md`；方案沉淀到 spec，并把入口补进 `plan.md`。

## 文件组织建议（建议照抄）
- `plan.md`：项目/模块的“导航索引”（常用入口链接 + 当前决策）
- `docs/specs/*.md`：机制/协议/策略说明（含验收/回滚）
- `docs/research/*.md`：全网/GitHub 大搜产物（可追溯链接 + 结论）
- `debug.md`：常见问题排查（症状→原因→解决→回滚）

---

## 交互/输出约定（给 AI 的硬规则）
- **语言**：默认简体中文。
- **结论先行**：先给结论与下一步，再给细节。
- **别猜上下文**：看不到就明说需要什么（文件路径/截图/日志/权限）。
- **一次只问一个关键澄清**：除非缺口太多，澄清最多 5 问。
- **实现前先计划**：列步骤/影响文件/风险/验证点；用户确认后再动手。
